# Configuration

All configuration lives in `fdl.toml`. Secrets are handled via `${VAR}` environment variable expansion.

## Concepts

`fdl.toml` has three orthogonal sections:

- `[metadata]` — the live catalog DB where writes land
- `[data]` — the Parquet file storage location
- `[publishes.<name>]` — zero or more frozen-snapshot destinations

## fdl config

Read and write `fdl.toml` from the command line:

```bash
# Set a value
fdl config metadata.url 'postgres://${PG_HOST}/fdl?schema=mart'

# Get a value
fdl config metadata.url

# List all settings
fdl config
```

## fdl.toml

Located at the project root and tracked in Git. Auto-generated by `fdl init`.

```toml
name = "my_dataset"
command = "python main.py"

[metadata]
url = "${FDL_METADATA_URL}"            # sqlite:// or postgres://
schema = "my_dataset"                  # optional METADATA_SCHEMA for postgres

[data]
url = "${FDL_DATA_URL}"                # local path or s3://
s3_endpoint = "${FDL_S3_ENDPOINT}"
s3_access_key_id = "${FDL_S3_ACCESS_KEY_ID}"
s3_secret_access_key = "${FDL_S3_SECRET_ACCESS_KEY}"

[publishes.public]
url = "${FDL_PUBLISH_URL}"             # s3:// or local path
public_url = "${FDL_PUBLIC_URL}"       # absolute URL baked into the frozen catalog
```

### Keys

| Key | Description |
|---|---|
| `name` | Datasource name |
| `command` | Default pipeline command for `fdl run` |
| `metadata.url` | Live catalog URL (`sqlite:///` or `postgres://`) |
| `metadata.schema` | Optional METADATA_SCHEMA for postgres backends |
| `data.url` | Parquet storage URL (local path or `s3://`) |
| `data.s3_endpoint` / `s3_access_key_id` / `s3_secret_access_key` | S3 credentials for data storage |
| `publishes.<name>.url` | Publish destination (`s3://` or local path) |
| `publishes.<name>.public_url` | Absolute URL baked into the frozen catalog's `DATA_PATH` (defaults to `data.url`) |
| `publishes.<name>.command` | Per-publish override of top-level `command` |

All values support `${VAR}` environment variable expansion. Set actual values via [direnv](https://direnv.net/) (`.envrc`), CI secrets, or shell exports.

## SQLite vs PostgreSQL

Pick the metadata backend based on how many writers you need:

- **SQLite** — single host, single writer. Good for individual development and CI/CD runs where each run is ephemeral and coordinated externally (e.g. GitHub Actions `concurrency:`).
- **PostgreSQL** — multiple writers across hosts, shared live catalog. Required for Dagster parallel runs, multi-developer projects, and any setup where writes can overlap.

Migrating between backends uses DuckLake's built-in `COPY FROM DATABASE`:

```sql
ATTACH 'ducklake:sqlite:.fdl/ducklake.sqlite' AS sqlite_lake;
ATTACH 'ducklake:postgres:dbname=ducklake host=...' AS pg_lake (DATA_PATH 's3://bucket/data');
COPY FROM DATABASE sqlite_lake TO pg_lake;
```

Then edit `metadata.url` to the new backend.

## Environment separation

Use `${VAR}` expansion and external environment variables to switch environments. There is no `--profile` flag or environment-specific config file; fdl relies on the [12-Factor](https://12factor.net/config) convention.

```bash
# .envrc (local development)
export FDL_METADATA_URL="sqlite:///./.fdl/ducklake.sqlite"
export FDL_DATA_URL="./.fdl/data"
export FDL_PUBLISH_URL="./dist"

# CI secrets (production)
FDL_METADATA_URL="postgres://...@db.internal/ducklake"
FDL_DATA_URL="s3://prod-bucket/my_dataset"
FDL_PUBLISH_URL="s3://public-bucket/my_dataset"
```
